# TCMAgent

一个面向 **中医问诊场景** 的、基于 **Neo4j 医学知识图谱 + deepagents + deep-agents-ui** 的 **收敛式 Agent 系统**。

本项目的目标不是做一个“泛聊天问诊机器人”，而是构建一个：

- **以医学知识为约束**
- **以安全治理为前提**
- **以动态追问收敛为核心**
- **以证据链可追踪为要求**
- **以工程化可扩展为落地方向**

的中医问诊 Agent 平台。

---

## 1. 项目定位

TCMAgent 聚焦于“**有迹可循的收敛式问诊**”。

它不采用死板的固定问卷，也不把“下一问”完全交给大模型自由发挥，而是采用：

- **安全必问**：先处理红旗征象、特殊人群、用药史等高风险信息
- **图谱驱动动态追问**：基于 Neo4j 中的疾病、症状、病机、证型及其关系，动态选择最有区分度的问题
- **LLM 负责表达与总结**：让模型负责自然交互、归纳总结、结构化解释
- **工具与状态机负责约束**：让流程、风控、状态更新和证据记录可控、可查、可审计

一句话总结：

> **TCMAgent 是一个“医学知识图谱驱动 + deepagents 编排 + 安全优先”的中医问诊 Agent 系统。**

## 2. 核心设计目标

### 2.1 问得少，但更有效
不是把所有槽位都问一遍，而是围绕当前候选空间，优先问最能帮助收敛的问题。

### 2.2 每一步都有依据
每轮追问都应能回答：
- 为什么问这个问题？
- 它要区分哪些候选疾病 / 证型 / 病机？
- 用户的回答如何影响当前判断？

### 2.3 风险优先于智能
无论推理多强，只要命中高风险路径，都必须优先中断普通流程，进入安全复核、转人工或转线下。

### 2.4 可解释、可追踪、可审计
系统不仅输出结果，还要保留：
- 候选生成依据
- 追问选择依据
- 风险判断依据
- 结构化状态变化
- 工具调用与决策轨迹

### 2.5 工程上可持续扩展
第一阶段先做稳导诊、问诊、收敛与风控；后续再扩展辨证、调理建议、随访和更高风险能力。

## 3. 设计原则

### 3.1 不是固定问卷，而是动态收敛
问诊过程不应是固定脚本顺序追问，而应根据当前事实和候选空间不断重算“下一问”。

### 3.2 不是全规则，也不是全模型
- **知识图谱**：负责医学关系、候选空间、问题区分度、证据路径
- **Agent / LLM**：负责自然语言交互、意图理解、信息归纳、阶段总结
- **工具层**：负责状态更新、图谱查询、风控与审计

### 3.3 安全底线必须工具化
所有高风险逻辑都应下沉到明确的工具和状态约束，而不是散落在 prompt 中。

### 3.4 状态优先于对话历史
系统核心不应是“聊天记录”，而应是结构化的 `case_state`。

### 3.5 患者可见输出与内部推理输出必须分层
- 患者侧：自然、保守、可理解
- 内部侧：结构化、可追踪、带证据链

## 4. 技术栈

本项目计划采用以下核心技术：

- **Python**
- **uv**：项目与依赖管理
- **deepagents**：Agent 编排与能力治理
- **deep-agents-ui**：交互层与可视化承载

- **LangGraph / LangChain 生态**：推理与工作流编排能力

## 5. 总体架构概览

系统采用五层结构：

### 5.1 交互层
由 `deep-agents-ui` 承载：
- 患者聊天界面
- 阶段性摘要
- 风险提示
- 内部审核视图
- 证据链展示

### 5.2 编排层
由 `clinical-supervisor` 负责：
- 读取病例状态
- 决定下一阶段动作
- 调用相应 agent / tool
- 控制中断与升级

### 5.3 能力层
第一阶段的核心能力包括：
- `triage-agent`
- `intake-agent`
- `safety-agent`
- `graph-reasoner`（建议先做成工具集合 / reasoning service）

### 5.4 知识层
由 Neo4j 承载：
- 症状
- 疾病
- 证型
- 病机
- 问题节点
- 特殊人群
- 红旗征象
- 禁忌关系
- 证据路径

### 5.5 状态与治理层
核心对象为 `case_state`，负责管理：
- 结构化问诊信息
- 当前候选集合
- 风险判断结果
- 问题记录
- 证据链
- 流程阶段
- 审计轨迹

## 6. 核心角色划分

### 6.1 `clinical-supervisor`
系统总调度，不直接承担深度医学推理。

职责：
- 控制流程阶段
- 汇总结果
- 协调其他能力
- 保证安全检查不被跳过

### 6.2 `triage-agent`
导诊与初筛代理。

职责：
- 判断是否适合线上继续
- 初步识别红旗症状
- 标记特殊人群
- 给出推荐路径

### 6.3 `intake-agent`
自然语言问诊执行者。

职责：
- 以自然、简洁的方式提问
- 解析患者回答
- 归一化患者描述
- 更新结构化事实

> 它的职责是“怎么问”和“怎么整理回答”，而不是“自由决定问什么”。

### 6.4 `graph-reasoner`
图谱驱动的收敛引擎。

职责：
- 基于当前事实生成候选疾病 / 证型 / 病机集合
- 从图谱中选择最有区分度的下一问
- 构建证据路径
- 更新候选排序
- 判断继续追问还是停止

### 6.5 `safety-agent`
横切式安全控制代理。

职责：
- 红旗复核
- 特殊人群风险判断
- 禁忌检查
- 是否转人工 / 转线下
- 给出是否允许继续的结论

## 7. 问诊机制：双轨制

### 7.1 轨道 A：安全必问
这部分问题必须优先确认，例如：
- 胸痛、呼吸困难、意识异常
- 高热、抽搐、持续呕吐、出血
- 孕期、儿童、高龄
- 重大基础病
- 正在用药
- 已有明确危险信号

特点：
- 固定约束
- 工具化执行
- 不交给模型自由裁量

### 7.2 轨道 B：图谱驱动动态追问
在安全底线之外，系统根据当前图谱候选空间，动态选择“信息增益最大”的下一问。

例如：
- 不是机械追问“口渴吗”
- 而是在当前候选 A / B / C 的分歧点上，优先问能帮助区分它们的问题

这样问诊就能做到：
- 不机械
- 有方向
- 可解释
- 可收敛

## 8. Neo4j 在系统中的作用

Neo4j 不是普通知识库，而是本项目的医学推理底座。

它将承担四类核心职责：

### 8.1 知识承载
存储医学节点与关系：
- 疾病
- 症状
- 证型
- 病机
- 观察项
- 红旗信号
- 禁忌
- 问题节点
- 路径关系

### 8.2 候选生成
根据主诉、已知事实、特殊人群标签，生成第一版候选集合。

### 8.3 问题选择
选择最能帮助收敛的下一问，而不是简单按模板追问。

### 8.4 证据链记录
保留每次追问与每次判断背后的图谱依据，满足解释与审计要求。

更详细的图谱设计见：
- [`docs/neo4j-schema.md`](docs/neo4j-schema.md)

## 9. `case_state`：系统的核心状态对象

TCMAgent 不是依赖长对话历史驱动，而是依赖结构化病例状态驱动。

建议的核心内容包括：

- 基础资料
- 主诉
- 已确认事实
- 缺失关键事实
- 冲突信息
- 已问问题
- 候选疾病 / 证型 / 病机
- 当前推荐下一问
- 风险标签
- 分诊等级
- 收敛程度
- 审核状态
- 审计轨迹

两个关键评分：
- `intake_completeness_score`：基础事实完整度
- `convergence_score`：候选空间收敛程度

> 这两个分数不能混为一谈。  
> 信息可以很多，但判断仍然发散；信息也可能不算全，但候选已足够收敛。

## 10. 第一阶段 MVP 边界

### 10.1 必做
第一版建议只做以下能力：

- `uv` 管理项目
- `deepagents` 编排主流程
- `deep-agents-ui` 作为交互层
- Neo4j 图谱接入
- `clinical-supervisor`
- `triage-agent`
- `intake-agent`
- `safety-agent`
- 图谱驱动的下一问选择
- 问诊状态与证据链保存

### 10.2 暂不做
第一阶段建议明确不做：

- 自动处方
- 患者可见的强诊断结论
- 高风险治疗建议自动化
- 复杂长期随访
- 完整病机评分系统
- 大规模个性化治疗推荐
- 多模态深度舌诊

## 13. 当前最重要的工程判断

本项目最关键的设计判断是：

> **问诊的“下一问选择权”不要交给纯 LLM，而要交给图谱推理层。**

LLM 可以负责：
- 改写问题
- 柔化语气
- 理解患者自然语言
- 归纳阶段总结
- 生成面向患者的表达

但不应让它在没有医学结构约束的前提下自由决定问诊路径。

这也是 TCMAgent 与普通问诊 Bot 的根本区别。

## 15. 当前结论

TCMAgent 将采用：

- **安全必问 + 图谱驱动动态追问** 的双轨问诊机制
- **Neo4j** 管理医学知识、关系与证据路径
- **deepagents** 负责流程编排与能力治理
- **deep-agents-ui** 承载患者与内部视图
- **结构化 `case_state`** 作为系统核心状态对象

最终目标是构建一个：

> **可追踪、可解释、逐步收敛、安全优先的中医问诊 Agent 系统。**
